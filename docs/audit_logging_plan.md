# Audit Logging & Database Integration Plan

This document outlines the strategy for implementing centralized audit logging and database integration for the BFSI project agents (`Intake`, `KYC`, `Decisioning`, `Disbursement`). All agents will connect to a unified Aiven Postgres database.

## 1. Database Connection Strategy

All agents will share the same Aiven Postgres instance currently used by the `IntakeAgent`.

## 2. Table Schemas

### 2.1 Central Audit Logs Table
This table captures every LangGraph node execution across all agents.

```sql
CREATE TABLE audit_logs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    application_id VARCHAR(50) NOT NULL,
    agent_name VARCHAR(50) NOT NULL, -- 'intake', 'kyc', etc.
    node_name VARCHAR(100) NOT NULL, -- 'ssn_node', 'credit_score_node', etc.
    input_state JSONB,               -- State before node execution
    output_state JSONB,              -- State after node execution
    status VARCHAR(20),              -- 'SUCCESS', 'FAILURE'
    error_message TEXT,
    execution_time_ms INTEGER,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

### 2.2 Agent-Specific Storable Returns

#### Intake Agent (`loan_applications`)
Stores the primary source of truth for a loan request.
- `application_id` (PK)
- `user_details` (JSONB)
- `requested_amount` (DECIMAL)
- `status` (VARCHAR)

#### KYC Agent (`kyc_results`)
Stores detailed outcomes of the KYC pipeline.
- `application_id` (FK)
- `ssn_verification` (JSONB): Result from `ssn_node`.
- `address_verification` (JSONB): Details from `address_node`.
- `aml_check` (JSONB): Status from `aml_node`.
- `face_liveness` (JSONB): Score and artifacts from `face_node`.
- `overall_risk_score` (DECIMAL): From `risk_aggregator_node`.

#### Decisioning Agent (`underwriting_decisions`)
Captures the risk evaluation and final credit decision.
- `application_id` (FK)
- `credit_score` (INTEGER)
- `risk_factors` (JSONB): Individual scores from `credit_score`, `utilization`, `exposure` nodes.
- `llm_justification` (TEXT): Detailed reasoning from `decision_llm_node`.
- `final_decision` (VARCHAR): `APPROVED`, `REJECTED`, `COUNTER_OFFER`.
- `counter_offer_details` (JSONB): If applicable.

#### Disbursement Agent (`disbursement_records`)
Records the final financial execution.
- `application_id` (FK)
- `payment_schedule` (JSONB): Generated by `generate_schedule_node`.
- `transaction_id` (VARCHAR): From `execute_transfer_node`.
- `transfer_status` (VARCHAR): `SUCCESS`, `PENDING`, `FAILED`.
- `receipt_url` (VARCHAR): Link or reference.

## 3. Wiring the System

### 3.1 The Audit Decorator
To automate logging for every node without boilerplate, we will implement an `@audit_node` decorator in a shared utility or each agent's `utils` package.

```python
def audit_node(agent_name: str):
    def decorator(func):
        @wraps(func)
        async def wrapper(state: dict, *args, **kwargs):
            start_time = time.time()
            input_copy = json.loads(json.dumps(state)) # Deep copy for safety
            try:
                result = await func(state, *args, **kwargs)
                status = "SUCCESS"
                error = None
                return result
            except Exception as e:
                status = "FAILURE"
                error = str(e)
                raise
            finally:
                duration = int((time.time() - start_time) * 1000)
                await save_audit_log(
                    application_id=state.get("application_id"),
                    agent_name=agent_name,
                    node_name=func.__name__,
                    input_state=input_copy,
                    output_state=result if status == "SUCCESS" else None,
                    status=status,
                    error_message=error,
                    execution_time_ms=duration
                )
        return wrapper
    return decorator
```

### 3.2 Database Persistence
- **Alembic**: Each agent will have its own `alembic` setup pointing to the same DB but managing its own subset of tables (or a shared migration strategy).
- **Repositories**: Agents will use Repository patterns to persist "Storable Returns" at the end of their respective workflows or at critical nodes.
